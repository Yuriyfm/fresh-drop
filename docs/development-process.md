# Development Process

## Подход

Разработка должна быть итеративной, маленькими шагами.

Одна итерация должна давать проверяемый результат.

## Размер задач

Предпочтительный размер задачи:

- один небольшой feature;
- один bugfix;
- один refactoring;
- один тестовый слой;
- одна spec-правка.

Не объединять в одной итерации:

- изменение архитектуры;
- новую функциональность;
- массовый рефакторинг;
- изменение UI;
- изменение API-контракта.

## Перед началом задачи

Перед изменениями агент должен:

1. Понять цель задачи.
2. Найти релевантный spec.
3. Если spec отсутствует — предложить или создать минимальный spec.
4. Кратко описать план.
5. Не открывать лишние документы.

## Во время задачи

Правила:

- менять минимальное количество файлов;
- не трогать нерелевантный код;
- не делать “улучшения заодно”;
- не добавлять зависимости без причины;
- не менять публичные контракты без обновления spec;
- не смешивать refactoring и feature-разработку.

## После задачи

Агент должен вернуть краткий отчёт:

- что сделано;
- какие файлы изменены;
- какие тесты добавлены или обновлены;
- как запустить проверку;
- есть ли известные ограничения.

## Definition of Done

Задача считается завершённой, если:

- поведение соответствует spec;
- код проходит типизацию;
- добавлены или обновлены тесты;
- нет очевидного мёртвого кода;
- нет лишних зависимостей;
- UI не показывает пользователю неподтверждённые данные как точные;
- ошибки API обработаны;
- есть понятный способ проверить результат.

## Правило остановки

Если во время реализации выяснилось, что задача требует большого изменения архитектуры, агент должен остановиться и предложить маленький следующий шаг.

Не надо переписывать проект целиком.
