# Tech and Architecture

## Назначение

Этот документ описывает базовый стек и архитектурные правила MVP.

Не открывай этот файл без необходимости. Используй его, если задача касается структуры проекта, выбора библиотеки, API-слоя, доменной логики, тестов или организации файлов.

## Базовый стек

Основной стек MVP:

- TypeScript;
- React;
- Vite;
- Spotify Web API;
- Vitest для unit-тестов;
- Testing Library для UI-тестов, если нужен тест UI;
- CSS Modules или простой CSS, если отдельный UI-kit не выбран.

Если в проекте уже выбран другой стек, фактический код проекта имеет приоритет над этим документом.

## Главный архитектурный принцип

UI не должен напрямую знать детали Spotify API.

Данные должны проходить путь:

```text
Spotify API DTO
  -> adapter / mapper
  -> domain model
  -> filters / business logic
  -> UI
```

## Data layer

MVP использует собственную БД как временный индекс свежих релизов.

Spotify API используется как источник данных для периодической синхронизации, а не как основной runtime-поиск для пользователя.

Основной поток данных:

```text
Spotify API
  -> sync job / ingestion
  -> database
  -> backend API
  -> frontend
```

Frontend не должен напрямую ходить в Spotify API для основного поиска релизов.

Frontend обращается к backend API приложения, а backend API читает данные из БД.

## Database

Для MVP используется PostgreSQL.

БД хранит только свежие релизы в пределах актуального окна, например за последний месяц.

Данные старше актуального периода должны удаляться или исключаться из пользовательского поиска.

## Backend responsibilities

Backend отвечает за:

* синхронизацию данных из Spotify API;
* сохранение релизов и артистов в БД;
* дедупликацию релизов;
* применение серверных фильтров;
* пагинацию;
* обработку ошибок;
* отдачу нормализованных данных frontend-приложению.

## Frontend responsibilities

Frontend отвечает за:

* отображение списка релизов;
* управление фильтрами;
* loading / empty / error states;
* отображение карточек релизов.

Frontend не должен знать структуру Spotify API DTO.

## Текущий технический каркас

Точка входа приложения:

- `src/main.tsx` — подключает React-приложение к DOM;
- `src/App.tsx` — текущий MVP UI и временные demo-данные.

Слои MVP:

- `src/domain/` — доменная модель релиза и бизнес-логика фильтрации;
- `src/spotify/` — DTO Spotify API и мапперы из Spotify-данных в доменную модель;
- `src/test/` — общий test setup;
- слой хранения/кэша пока не реализован, потому что MVP ещё не загружает реальные данные. Когда появится загрузка из Spotify API, кэш должен быть отдельным слоем между Spotify adapter и доменной логикой, без прямой привязки к UI.

Базовые команды разработки:

```text
TMPDIR=/tmp yarn dev --host 0.0.0.0
TMPDIR=/tmp yarn typecheck
TMPDIR=/tmp yarn test:run
TMPDIR=/tmp yarn build
```

`TMPDIR=/tmp` нужен для WSL-окружения, где стандартная временная директория Yarn может указывать на read-only Windows-путь.
