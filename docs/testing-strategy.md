# Testing Strategy

## Цель тестов

Тесты нужны не ради формального покрытия, а чтобы экономить ресурсы агента и безопасно двигаться маленькими итерациями.

Тесты должны быстро показывать, что основная бизнес-логика не сломалась.

## Приоритет тестов

### 1. Unit-тесты доменной логики

Покрывать:

- фильтрацию релизов по периоду;
- фильтрацию по жанру;
- фильтрацию по стране;
- фильтрацию по типу релиза;
- сортировку или фильтрацию по popularity;
- нормализацию дат;
- объединение жанров артистов;
- обработку неизвестной страны.

### 2. Тесты адаптеров данных

Покрывать преобразование данных Spotify API в внутренние модели приложения.

Важно тестировать:

- отсутствующие поля;
- пустые массивы;
- неизвестный тип релиза;
- неполную дату релиза;
- несколько артистов;
- отсутствие genres у артиста.

### 3. Интеграционные тесты

Добавлять только для ключевых сценариев.

Например:

- загрузка релизов;
- обогащение артистами;
- применение фильтров;
- возврат результата для UI.

### 4. UI-тесты

На MVP этапе минимально.

Проверять только критичные состояния:

- loading;
- empty state;
- error state;
- список релизов;
- изменение фильтра.

## Что не тестировать в MVP

- внутренности внешних библиотек;
- точный внешний вид карточек;
- реальные ответы Spotify API в unit-тестах;
- нестабильные сетевые сценарии без моков;
- сложные e2e-сценарии.

## Правила для агента

Если добавляешь новую бизнес-логику — добавь unit-тест.

Если меняешь адаптер Spotify API — добавь тест на пример ответа API.

Если исправляешь баг — сначала добавь тест, который воспроизводит баг, затем исправь код.

Если тесты требуют слишком много setup-кода — предложи упростить архитектуру.
