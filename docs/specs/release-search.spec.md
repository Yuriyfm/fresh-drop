# Spec: Release Search

## Назначение

Спецификация описывает поведение поиска свежих релизов Spotify для MVP.

Цель функции — дать пользователю возможность найти свежие релизы за последние 7 дней, 14 дней или 1 месяц и отфильтровать их по жанру, стране, популярности и типу релиза.

## Контекст MVP

MVP работает без авторизации пользователя через Spotify.

Основной источник данных — Spotify API.

Базовый подход к получению свежих релизов:

- использовать поиск Spotify с `tag:new`;
- получать релизы;
- фильтровать их по дате выхода;
- обогащать релизы данными артистов;
- применять пользовательские фильтры.

Если возможности Spotify API отличаются от предположений этой спецификации, сначала обнови spec, затем меняй реализацию.

## Основной пользовательский сценарий

1. Пользователь открывает страницу поиска свежих релизов.
2. Приложение загружает список свежих релизов.
3. Пользователь выбирает период:
   - 7 дней;
   - 14 дней;
   - 1 месяц.
4. Пользователь может применить фильтры:
   - жанр;
   - страна;
   - популярность;
   - тип релиза.
5. Приложение показывает список релизов, соответствующих выбранным фильтрам.
6. Если релизов нет, приложение показывает empty state.
7. Если данные не удалось загрузить, приложение показывает error state.

## Модель данных

### ReleaseType

```ts
type ReleaseType = 'single' | 'album' | 'compilation' | 'unknown';
```

### ReleaseDatePrecision

```ts
type ReleaseDatePrecision = 'year' | 'month' | 'day' | 'unknown';
```

### ArtistSummary

```ts
type ArtistSummary = {
  id: string;
  name: string;
  genres: string[];
  country: string | 'unknown';
  popularity: number | null;
};
```

### Release

```ts
type Release = {
  id: string;
  spotifyUrl: string | null;
  coverUrl: string | null;
  title: string;
  artists: ArtistSummary[];
  primaryArtist: ArtistSummary | null;
  type: ReleaseType;
  releaseDate: string;
  releaseDatePrecision: ReleaseDatePrecision;
  genres: string[];
  country: string | 'unknown';
  popularity: number | null;
};
```

## Правила нормализации данных

### ID

`id` должен быть стабильным Spotify ID релиза.

Если ID отсутствует, такой релиз не должен попадать в финальный список.

### Spotify URL

`spotifyUrl` должен вести на релиз в Spotify.

Если URL недоступен, использовать `null`.

### Cover URL

`coverUrl` должен содержать URL обложки релиза.

Если обложка недоступна, использовать `null`.

### Название

`title` должно содержать название релиза.

Если название отсутствует, релиз не должен попадать в финальный список.

### Артисты

`artists` должен содержать список артистов релиза.

Если артисты отсутствуют, использовать пустой массив.

`primaryArtist` — первый артист из списка, если он есть.

Если список артистов пустой, `primaryArtist = null`.

### Тип релиза

Тип релиза должен быть одним из:

- `single`;
- `album`;
- `compilation`;
- `unknown`.

Если Spotify API возвращает неизвестное значение, использовать `unknown`.

### Дата релиза

`releaseDate` хранится строкой в формате, полученном из Spotify API.

`releaseDatePrecision` показывает точность даты.

Для фильтрации по последним 7 или 14 дням релиз должен иметь точность `day`.

Если точность даты ниже `day`, такой релиз нельзя надёжно использовать в дневных фильтрах.

### Жанры

`genres` формируется из жанров:

- главного артиста;
- всех артистов релиза.

Правила нормализации жанров:

- привести к lowercase;
- удалить пробелы в начале и конце;
- удалить пустые значения;
- удалить дубли.

Пример:

```ts
['Indie Rock', ' indie rock ', 'Pop']
// becomes
['indie rock', 'pop']
```

### Страна

`country` — страна артиста или релиза, если её можно определить.

Если страна не может быть определена надёжно, использовать `unknown`.

Нельзя показывать `unknown` как точную страну пользователя или артиста.

### Popularity

`popularity` — число от 0 до 100 или `null`.

Для MVP можно использовать:

- popularity главного артиста;
- или popularity релиза, если Spotify API предоставляет такое поле стабильно.

Выбранный источник popularity должен быть зафиксирован в коде и тестах.

Если popularity недоступна, использовать `null`.

## Фильтры

### Фильтр по периоду

Поддерживаемые значения:

```ts
type ReleasePeriod = '7d' | '14d' | '1m';
```

Правила:

- `7d` — релизы за последние 7 календарных дней.
- `14d` — релизы за последние 14 календарных дней.
- `1m` — релизы за последний месяц.

Дата сравнивается относительно текущей даты приложения.

Для `7d` и `14d` учитывать только релизы с `releaseDatePrecision = 'day'`.

Для `1m` предпочтительно также использовать `releaseDatePrecision = 'day'`.

Если в реализации принято решение включать в `1m` релизы с точностью `month`, это должно быть явно покрыто тестами.

### Фильтр по жанру

Релиз проходит фильтр по жанру, если выбранный жанр есть в `release.genres`.

Сравнение жанров должно быть case-insensitive.

Если жанр не выбран, фильтр по жанру не применяется.

### Фильтр по стране

Релиз проходит фильтр по стране, если `release.country` совпадает с выбранной страной.

Если страна не выбрана, фильтр по стране не применяется.

Если страна релиза `unknown`, релиз должен попадать только в фильтр `unknown`, если такой фильтр явно есть в UI.

### Фильтр по типу релиза

Поддерживаемые значения:

```ts
type ReleaseTypeFilter = 'all' | 'single' | 'album' | 'compilation';
```

Правила:

- `all` — показывать все типы релизов.
- `single` — показывать только синглы.
- `album` — показывать только альбомы.
- `compilation` — показывать только compilation-релизы.

Если Spotify API не позволяет стабильно получать синглы через выбранный endpoint, UI не должен обещать полный список синглов.

### Фильтр по популярности

Поддерживаемые значения:

```ts
type PopularityFilter = 'all' | 'popular' | 'less-known';
```

Первичная логика MVP:

- `all` — показывать все релизы.
- `popular` — показывать релизы, где `popularity >= 60`.
- `less-known` — показывать релизы, где `popularity < 60`.

Если `popularity = null`:

- релиз не считается `popular`;
- релиз может попадать в `all`;
- для `less-known` поведение должно быть явно определено в тестах.

Базовое правило для MVP:

```text
null popularity попадает только в all.
```

## Сортировка

Базовая сортировка MVP:

1. Более свежие релизы выше.
2. При одинаковой дате — релизы с большей popularity выше.
3. При отсутствии popularity — релиз ниже релизов с известной popularity.

Если дата релиза неизвестна или недостаточно точна, релиз должен быть ниже релизов с точной датой.

## Состояния UI

### Loading state

Показывается во время загрузки данных.

Во время загрузки не показывать пустой список как финальный результат.

### Empty state

Показывается, если загрузка прошла успешно, но релизов по выбранным фильтрам нет.

Текст может быть таким:

```text
Свежих релизов по выбранным фильтрам не найдено.
Попробуйте изменить период, жанр или страну.
```

### Error state

Показывается, если данные не удалось загрузить.

Правила:

- не показывать ошибку как empty state;
- дать пользователю возможность повторить запрос;
- не показывать технические детали ошибки без необходимости.

Пример текста:

```text
Не удалось загрузить свежие релизы.
Попробуйте повторить запрос позже.
```

## Карточка релиза

Карточка релиза должна показывать:

- обложку;
- название релиза;
- артиста или артистов;
- тип релиза;
- дату выхода;
- жанры;
- страну или `unknown`;
- popularity, если доступно.

Если поле неизвестно, UI должен показывать это аккуратно.

Например:

- страна неизвестна;
- жанры не указаны;
- popularity недоступна.

## Ограничения MVP

MVP не обязан поддерживать:

- Spotify login;
- персональные рекомендации;
- сохранение релизов в Spotify;
- создание плейлистов;
- лайки и закладки;
- историю просмотров;
- социальные функции;
- сложную рекомендательную систему.

## Обработка ошибок данных

Реализация должна корректно обрабатывать:

- пустой ответ Spotify API;
- отсутствие артистов;
- отсутствие genres;
- отсутствие images;
- отсутствие external_urls;
- неизвестный release type;
- неполную дату релиза;
- rate limit;
- сетевую ошибку;
- ошибку авторизации приложения.

## Требования к тестам

Unit-тестами должны быть покрыты:

- фильтрация по периоду;
- фильтрация по жанру;
- фильтрация по стране;
- фильтрация по типу релиза;
- фильтрация по popularity;
- сортировка релизов;
- нормализация жанров;
- обработка `unknown` страны;
- обработка `null` popularity;
- обработка неполной даты релиза.

Тестами адаптера Spotify API должны быть покрыты:

- нормальный ответ API;
- релиз без обложки;
- релиз без артистов;
- артист без genres;
- неизвестный тип релиза;
- release date с точностью `year`;
- release date с точностью `month`;
- release date с точностью `day`.

## Acceptance Criteria

Функция считается готовой, если:

- пользователь может выбрать период `7d`, `14d` или `1m`;
- пользователь может фильтровать релизы по жанру;
- пользователь может фильтровать релизы по стране;
- пользователь может фильтровать релизы по типу;
- пользователь может фильтровать релизы по популярности;
- список релизов сортируется по свежести и popularity;
- карточка релиза показывает основные поля;
- loading state реализован;
- empty state реализован;
- error state реализован;
- бизнес-логика покрыта unit-тестами;
- адаптер Spotify API покрыт тестами;
- ограничения Spotify API не скрываются от пользователя;
- поведение соответствует этой спецификации.

## Правило изменения спецификации

Если во время разработки выяснилось, что Spotify API не позволяет реализовать часть поведения так, как описано здесь, нужно:

1. Обновить эту спецификацию.
2. Зафиксировать новое поведение.
3. Обновить тесты.
4. Только после этого менять реализацию.
