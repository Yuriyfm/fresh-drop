# Spec: Product MVP

## Назначение

Этот документ фиксирует базовый контракт MVP Fresh Drop.

MVP должен проверить один главный вопрос:

```text
Может ли пользователь удобно найти свежую музыку, добавленную в Spotify за последние 7 дней, 14 дней или 1 месяц, используя фильтры по жанру, стране, популярности и типу релиза?
```

Если продуктовая идея, UX или код противоречат этому документу, сначала нужно обновить spec, затем менять реализацию.

## Пользовательская ценность

Пользователь хочет быстро увидеть свежие релизы Spotify, сузить список понятными фильтрами и открыть подходящий релиз в Spotify.

Приложение не является рекомендательной системой и не пытается предсказывать личный вкус пользователя.

## Первая версия MVP

В первую версию входят:

- список свежих релизов Spotify;
- выбор периода: `7d`, `14d`, `1m`;
- фильтр по одному или нескольким жанрам;
- фильтр по стране;
- фильтр по популярности;
- фильтр по типу релиза;
- сортировка по свежести и popularity;
- карточка релиза с основными данными;
- переход к релизу в Spotify;
- loading, empty и error states.

## Фильтры первой версии

Точные фильтры первой версии:

```ts
type ReleasePeriod = '7d' | '14d' | '1m';
type ReleaseTypeFilter = 'all' | 'single' | 'album' | 'compilation';
type ReleaseSort = 'newest' | 'oldest' | 'popular' | 'less-popular';
```

Фильтры по жанру и стране строятся по значениям, которые реально удалось получить из данных Spotify после нормализации.

Жанровый фильтр поддерживает выбор нескольких жанров с логикой `OR`, специальный вариант `без жанра` для релизов без жанров и приоритетный показ общих верхнеуровневых жанров, если по ним есть релизы.

Если значение неизвестно или ненадёжно, оно должно быть представлено как `unknown`, а не как предположение.

## Не входит в MVP

В MVP не входят:

- Spotify login пользователя;
- персональные рекомендации;
- лайки, закладки и избранное;
- сохранение релизов в Spotify;
- создание плейлистов;
- история просмотров;
- шаринг;
- социальные функции;
- внешние базы данных для обогащения артистов;
- сложная система определения страны артиста.

Эти функции можно добавлять только после отдельного решения и обновления spec.

## Контракт данных Spotify

Spotify API считается источником данных для MVP, но не все поля одинаково точны.

### Считаем точным

Точными считаются поля, которые Spotify API возвращает явно для релиза или артиста:

- Spotify ID релиза;
- название релиза;
- список артистов релиза;
- `external_urls.spotify`, если поле есть;
- изображения релиза, если поле есть;
- `album_type`, если endpoint возвращает его явно;
- `release_date` вместе с `release_date_precision`;
- жанры артиста, если они есть в ответе Artist API;
- popularity артиста, если она есть в ответе Artist API.

Даже точные поля могут отсутствовать. В этом случае применяются правила `null`, пустого массива или `unknown` из `release-search.spec.md`.

### Считаем приблизительным

Приблизительными считаются значения, которые MVP выводит из доступных данных:

- жанры релиза как объединение жанров артистов;
- популярность релиза как popularity главного артиста;
- страна релиза или артиста, если она выведена из market availability или других косвенных признаков;
- полнота выдачи свежих релизов через Search API с `tag:new`.

Такие значения можно использовать для фильтрации, но UI и тексты не должны обещать абсолютную полноту или точность.

### Показываем как unknown

Как `unknown` или пользовательский эквивалент "неизвестно" показываются:

- страна, если нет надёжного источника;
- тип релиза, если Spotify вернул неизвестное значение;
- точность даты, если она отсутствует или не распознана;
- любые данные, которые нельзя получить без предположений вне Spotify API.

`unknown` нельзя заменять догадкой ради заполнения интерфейса.

## Ограничения Spotify API для MVP

MVP явно принимает эти ограничения:

- Search API с `tag:new` может не быть полным каталогом всех новых релизов;
- доступность релизов может зависеть от market;
- страна артиста напрямую может быть недоступна;
- жанры обычно привязаны к артистам, а не к релизам;
- popularity может быть доступна не на том уровне данных, который нужен продукту;
- даты релизов могут иметь точность `year`, `month` или `day`;
- для периодов `7d` и `14d` нужны даты с точностью `day`;
- API может возвращать rate limit, сетевые ошибки или ошибки авторизации приложения.

Если во время реализации обнаружено новое ограничение API, нужно сначала обновить этот spec или `release-search.spec.md`.

## Критерии готовности MVP-контракта

MVP соответствует контракту, если:

- пользователь может искать свежие релизы по периодам `7d`, `14d`, `1m`;
- доступны фильтры первой версии;
- неизвестные данные не выдаются за точные;
- ограничения Spotify API отражены в поведении и UI;
- бизнес-логика поиска соответствует `release-search.spec.md`;
- новая бизнес-логика покрыта unit-тестами.
