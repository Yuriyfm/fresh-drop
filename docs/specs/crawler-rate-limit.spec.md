# Crawler Rate Limit Spec

## Назначение

Этот документ фиксирует поведение crawler при ограничениях Spotify Web API.

Используй его, если задача касается:

* плановой загрузки релизов;
* retry и backoff;
* обработки `429` и временных ошибок Spotify API;
* очереди `sync_tasks`.

## Проблема

Spotify не публикует фиксированный численный лимит для Search API.

Для Web API действует rate limit в rolling window 30 секунд, а при превышении лимита API возвращает `429 Too Many Requests` и может прислать `Retry-After`.

Crawler не должен продолжать агрессивно отправлять запросы после `429`, иначе очередь быстро заполняется ошибками и рост базы останавливается.

## Цели MVP

Crawler должен:

* продолжать плановые загрузки без ручного вмешательства;
* замедляться при rate limit;
* заранее снижать вероятность `429` за счёт осторожного pacing запросов;
* возвращать retryable-задачи обратно в очередь;
* не превращать временный rate limit в постоянную потерю задач.

## Правила

### 1. Retryable ошибки

Retryable ошибками для crawler считаются:

* `429 Too Many Requests`;
* временные `5xx` ошибки Spotify API;
* кратковременные сетевые ошибки.

Такие ошибки не должны оставлять задачу в терминальном состоянии, если лимит попыток не исчерпан.

### 2. Поведение при `429`

Если Spotify API возвращает `429`, crawler обязан:

* прочитать `Retry-After`, если он есть;
* использовать консервативный fallback backoff, если `Retry-After` отсутствует;
* трактовать `Retry-After` как время до следующего допустимого окна запросов;
* вернуть текущую задачу в очередь с будущим `next_run_at`;
* отложить остальные pending-задачи crawler до того же времени;
* прекратить обработку оставшейся части текущего batch в этом запуске.

### 3. Осторожный pacing запросов

Crawler и Spotify adapter не должны слать Web API запросы максимально быстро.

Минимальное правило для MVP:

* между HTTP-запросами к Spotify должен быть минимальный интервал;
* этот интервал задаётся конфигом и имеет безопасное значение по умолчанию `10` секунд;
* после `429` adapter не должен отправлять следующий Spotify запрос раньше окна, вычисленного из `Retry-After`.

### 4. Поведение очереди

Для retryable ошибок задача должна снова становиться доступной для claim после паузы.

Минимальное правило для MVP:

* retryable задача должна храниться в состоянии, которое выбирается обычным `claimPendingTasks`;
* `next_run_at` определяет, когда задача снова может быть взята в работу;
* терминальный `failed` допустим только для неретраибельной ошибки или после достижения лимита попыток.

### 5. Поведение batch

Crawler не должен продолжать отправлять новые запросы в том же batch после первого подтверждённого rate limit.

Если `429` произошёл внутри batch:

* текущий запуск завершает обработку batch раньше срока;
* уже не обработанные claimed-задачи возвращаются в очередь с паузой;
* запуск может завершиться с частичным результатом и понятной диагностикой в логах.
* если часть релизов уже была получена до `429`, они должны быть сохранены и учтены в статистике запуска;
* частичный результат не должен теряться только потому, что следующая страница или enrichment запрос упёрлись в rate limit.

### 6. Наблюдаемость

Логи crawler должны позволять понять:

* что запуск остановился из-за rate limit;
* какой backoff и какое время следующего окна запросов были выбраны;
* сколько задач было отложено или перенесено;
* сколько задач реально сохранили данные.
* сколько HTTP-запросов к Spotify было сделано за запуск и по каждой обработанной задаче.

## Ограничения MVP

В рамках MVP не требуется:

* вычислять точный глобальный quota budget Spotify;
* строить сложный distributed rate limiter;
* добавлять отдельную внешнюю очередь.

Достаточно консервативного backoff, корректного retry и безопасного поведения batch.
