Инструкции для AI-агентов при работе над MVP приложения поиска свежей музыки в Spotify.

## Главный принцип

Не перегружай контекст.

В каждом чате используй только этот файл и те документы из `docs/`, которые явно нужны для текущей задачи.

Не открывай все `.md` файлы подряд. Сначала определи тип задачи, затем открой только релевантные документы.

## Как работать

1. Не начинай писать код, пока не понял задачу и не проверил релевантную спецификацию.
2. Работай маленькими итерациями.
3. Не делай крупные архитектурные изменения без явной необходимости.
4. Не добавляй функциональность “на будущее”, если она не нужна для MVP.
5. Перед изменениями кратко опиши план.
6. После изменений кратко опиши:
    - что изменено;
    - какие файлы затронуты;
    - как проверить результат;
    - какие тесты добавлены или обновлены.

## Экономия токенов

- Не пересказывай длинные документы без необходимости.
- Не открывай нерелевантные файлы.
- Не анализируй весь проект, если задача касается одного модуля.
- Не переписывай большие файлы целиком, если достаточно точечного изменения.
- Не добавляй подробные комментарии в код без необходимости.
- Предпочитай короткие, конкретные ответы.

## Работа с контекстом

Экономь контекст. Не загружай все документы проекта без необходимости.

Перед началом задачи определи, какие файлы действительно нужны. Читай только релевантные спецификации и только нужные разделы, если этого достаточно.

* Для продуктового скоупа и границ MVP смотри `docs/specs/product.spec.md`.
* Для логики поиска свежих релизов смотри `docs/specs/release-search.spec.md`.
* Для UX/UI-задач сначала смотри `docs/specs/ux.summary.md`.
* Полный UX-документ `docs/specs/ux.spec.md` открывай только если краткой версии недостаточно.
* Для API-контрактов смотри `docs/specs/api.spec.md`, если такой файл есть.
* Для архитектурных решений смотри `docs/decisions/`.

Не вставляй содержимое спецификаций в чат целиком, если пользователь явно не просит. Кратко пересказывай только нужные выводы.

Если задача небольшая, сначала предложи минимальный план изменений и реализуй её маленьким шагом. Не пытайся за один проход менять много несвязанных частей приложения.

Перед реализацией проверь:

* соответствует ли изменение релевантной спецификации;
* не выходит ли оно за рамки MVP;
* нужны ли тесты или обновление существующих тестов;
* не добавляет ли изменение лишнюю сложность в UX или архитектуру.

После изменения кратко опиши:

* что изменено;
* какие файлы затронуты;
* как проверить результат;
* какие тесты запускались или почему не запускались.

## Стиль разработки

- TypeScript-first.
- Явные типы для публичных функций, API-адаптеров и доменных моделей.
- Простые функции лучше сложных абстракций.
- Дублирование допустимо на раннем MVP, если абстракция пока не очевидна.
- Бизнес-логику отделять от UI.
- Работу с внешними API изолировать в отдельных сервисах/адаптерах.

## Spec-driven development

Если задача меняет поведение продукта:

1. Сначала обнови или создай spec в `docs/specs/`.
2. Затем реализуй код.
3. Затем добавь или обнови тесты.
4. Затем проверь соответствие реализации spec-файлу.

Если spec противоречит коду — не угадывай. Сообщи о противоречии и предложи минимальное исправление.

## Тесты

Для новой бизнес-логики тесты обязательны.

Минимально покрывать:

- фильтрацию релизов;
- преобразование данных Spotify API в доменную модель;
- работу с датами релизов;
- типы релизов: album / single / compilation;
- обработку пустых данных и ошибок API.

Не добавляй тяжёлые e2e-тесты без явной необходимости.

## MVP-принципы

MVP должен отвечать на один главный вопрос:

Может ли пользователь удобно найти свежую музыку, добавленную в Spotify за последние 7 дней, 14 дней или 1 месяц, используя фильтры по жанру, стране, популярности и типу релиза?

Всё, что не помогает проверить этот вопрос, откладывается.

## UX-принципы проекта

Приложение проектируется как mobile-first веб-сервис для поиска свежих релизов Spotify.

Ключевые UX-принципы:

* Главная единица интерфейса — релиз.
* Главная цель пользователя — быстро найти подходящий релиз и открыть его в Spotify.
* Приложение не является рекомендательной системой.
* Интерфейс должен быть search-first: фильтры, сортировка, компактный список.
* Приоритет — удобство мобильных пользователей.
* Визуальный стиль ориентируется на Spotify: тёмная тема по умолчанию, музыкальный минимализм, зелёный акцент.
* Не добавляй избранное, авторизацию, персональные рекомендации, шаринг и другие не-MVP функции без явного решения.
* Для UX/UI-задач сначала используй `docs/specs/ux.summary.md`; полный `docs/specs/ux.spec.md` открывай только при необходимости.
